## 拿到服务器信息后做什么

按以下顺序推进：

1. 整理服务器信息。
2. 根据实际环境修改服务器配置方案。
3. 完成 Docker 部署文件。
4. 完成钉钉用户身份传递和白名单。
5. 在办公电脑上进行本地容器测试。
6. 由你或运维部署 mock 版本。
7. 进行权限、安全、异常恢复和端到端测试。
8. 小范围开放真实 DeepSeek API。
9. 编写验收记录和运维文档。

### 第一步：判断实际部署路线

拿到信息后，先整理成表格：

| 项目     | 实际信息             | 对方案的影响               |
| ------ | ---------------- | -------------------- |
| 操作系统   | 例如 Ubuntu 24.04  | 决定安装命令和目录            |
| Docker | 已安装/未安装/不允许      | 决定容器部署还是虚拟环境部署       |
| 网络     | 能否访问钉钉和 DeepSeek | 决定代理与出口白名单           |
| 入站端口   | 是否允许开放           | Stream 模式下可能无需开放业务端口 |
| 登录方式   | SSH/堡垒机/运维代部署    | 决定谁执行部署              |
| 服务器用途  | 专用/与其他项目共用       | 决定隔离强度               |
| 公司数据要求 | 是否允许发送外部模型       | 决定哪些功能可以使用 DeepSeek  |

如果服务器不允许 Docker，就改为：

```text
Python 虚拟环境 + systemd + OpenClaw + FastAPI
```

如果允许 Docker，优先采用：

```text
Docker Compose + OpenClaw + FastAPI
```

### 第二步：补充部署文件

需要补充：

```text
Dockerfile
compose.yaml
.dockerignore
.env.production.example
docs/deployment.md
docs/security.md
docs/test-and-acceptance.md
```

这些才是从“本地项目”变成“服务器项目”的关键成果。

## 用户分区应该怎么处理

这里的“用户分区”不是给每个钉钉员工创建一个 Linux 账号，而是分成三个层次。

### 1. 服务器账号隔离

服务器只需要少量运维账号：

| 账号    | 用途                    | 权限            |
| ----- | --------------------- | ------------- |
| 系统管理员 | 系统、网络和安全维护            | 可以使用受控的 sudo  |
| 部署账号  | 发布代码和启停服务             | 只能管理本项目       |
| 服务账号  | 运行 OpenClaw 和 FastAPI | 无 sudo、禁止交互登录 |
| 审计账号  | 查看日志                  | 只读，可选         |

公司员工不需要拥有服务器账号，他们只通过钉钉使用机器人。

如果最后由公司运维统一部署，你只需要在方案里给出推荐账号模型，不需要自己创建。

### 2. 钉钉用户权限隔离

这是当前最需要实现的部分。

推荐第一阶段只做简单白名单：

```env
ALLOWED_USER_IDS=user_id_1,user_id_2,user_id_3
ADMIN_USER_IDS=admin_user_id
```

处理流程：

```text
钉钉消息
→ OpenClaw 获取真实钉钉用户 ID
→ 调用 FastAPI 时传入 user_id
→ FastAPI 检查白名单
→ 允许或拒绝
→ 通过后才调用 DeepSeek
```

特别注意：不能让 OpenClaw 的调用脚本始终传固定的 `test_user`。必须确认传给 `/chat` 的 `user_id` 是当前发送者的真实钉钉用户标识。

第一阶段可以设计三个角色：

| 角色            | 权限                   |
| ------------- | -------------------- |
| `employee`    | 生成日报                 |
| `pilot_user`  | 使用待测试的新功能            |
| `agent_admin` | 管理白名单、查看脱敏状态         |
| 未知用户          | 不允许调用 DeepSeek 和公司工具 |

### 3. 会话和数据隔离

需要防止不同用户的上下文混在一起：

```text
用户A → 用户A自己的会话
用户B → 用户B自己的会话
```

会话标识可以按照下面的方式生成：

```text
source + group_id + user_id + session_id
```

例如：

```text
dingtalk:group_001:user_123:session_01
```

日志记录：

* `request_id`
* 脱敏后的 `user_id`
* 来源和群 ID
* 调用了什么功能
* 是否通过权限校验
* 是否调用 DeepSeek
* 成功、失败和耗时

日志中不要记录：

* DeepSeek API Key；
* 钉钉 Client Secret；
* 完整认证头；
* 不必要的公司敏感内容；
* 完整 `.env`。

## 权限功能的最低测试要求

完成用户白名单后，至少增加以下测试：

1. 白名单用户生成日报成功。
2. 非白名单用户被拒绝。
3. 缺少 `user_id` 时被拒绝。
4. 被拒绝的请求不调用 DeepSeek。
5. 不同用户的会话不会混合。
6. 管理员和普通员工权限不同。
7. 日志记录权限结果，但不泄露密钥。
8. OpenClaw 能把真实钉钉用户 ID 传入 FastAPI。

## 你接下来的具体任务

在等待服务器信息期间，可以先做两件事：

1. 检查 OpenClaw 的 `call_fastapi.py`，确认它是否传入真实钉钉用户 ID，而不是固定的 `test_user`。
2. 在 FastAPI 中实现最小白名单校验，并为它编写 pytest 测试。

服务器信息返回后，再开始编写 `Dockerfile` 和 `compose.yaml`。

你目前的项目路线可以概括为：

```text
功能闭环已完成
→ 用户权限
→ Docker 部署
→ 服务器 mock 测试
→ 安全与异常验收
→ 真实 API 小范围试运行
→ 再考虑知识库和其他功能
```

这比现在直接增加更多功能更合理，也更符合“公司服务器内部 AI Agent”的最终目标。
