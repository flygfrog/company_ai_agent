# 尧唐科技内部 AI Agent：服务器部署与安全配置方案

> 文档状态：规划稿（V0.1）  
> 更新日期：2026-07-15  
> 适用阶段：本地 MVP 已跑通，准备进入公司服务器部署  
> 待确认事项：服务器操作系统、网络环境、账号管理制度、数据敏感等级和公司运维要求

## 1. 项目目标

### 1.1 最终目标

在公司服务器上部署一个面向内部员工的 AI Agent，使员工能够通过钉钉机器人提交自然语言请求，由系统完成身份校验、意图识别、Agent 调度、大模型或内部工具调用，并将结果安全地返回钉钉。

目标链路如下：

```text
公司员工
  → 钉钉企业内部机器人
  → OpenClaw（消息接入与 Agent/Skill 调度）
  → FastAPI 业务后端
  → DeepSeek API / 公司知识库 / 受控工具
  → FastAPI 与 OpenClaw
  → 钉钉回复
```

### 1.2 第一阶段可交付目标

第一阶段不是建设一个“什么都能做”的全能智能体，而是交付一个安全、稳定、可审计的内部原型：

1. 钉钉中的授权员工可以使用机器人生成规范日报。
2. 未授权用户、未知意图和高风险操作会被明确拒绝或要求人工确认。
3. 服务能够在服务器重启或异常退出后自动恢复。
4. 管理员能够通过健康检查和日志定位一次请求的完整链路。
5. API Key、钉钉密钥和公司数据不进入代码仓库或普通业务日志。
6. 不同系统用户、钉钉用户和业务数据之间具备必要隔离。

### 1.3 暂不纳入第一阶段的内容

- 不开放任意 Shell 命令、任意文件读写或无限制网络访问。
- 不直接连接正式业务数据库执行写操作。
- 不让 Agent 自动发送通知、修改文件或审批业务，除非后续加入白名单和人工确认。
- 不在权限体系完成前导入公司敏感知识库。
- 不把 OpenClaw 的通用大模型回复等同于公司后端功能；正式功能必须经过 FastAPI、权限校验和日志链路。

## 3. 推荐部署架构

### 3.1 推荐方案

推荐使用 Linux 服务器和 Docker Compose 管理 OpenClaw、FastAPI 及后续依赖。若公司已有 Kubernetes、统一容器平台或运维规范，应优先服从公司现有标准，而不是单独引入新体系。

```text
钉钉 Stream（出站长连接）
        │
        ▼
OpenClaw 容器/服务
        │ 仅内部网络访问
        ▼
FastAPI 容器 :8000
        │
        ├── DeepSeek API（受限出站访问）
        ├── 日志目录（持久化）
        └── 后续知识库/数据库（独立权限）
```

采用钉钉 Stream 模式时，业务链路通常由服务器主动建立出站连接，因此 FastAPI 的 `8000` 端口不应直接暴露到公网。SSH 只允许公司指定网段、VPN 或堡垒机访问；只有确实需要外部 HTTP 回调时才开放 `443`。

### 3.2 服务边界

| 服务 | 责任 | 网络原则 | 运行身份 |
| --- | --- | --- | --- |
| OpenClaw | 接收钉钉消息、选择受控 Skill、返回消息 | 允许访问钉钉；只能访问内部 FastAPI | 非 root 服务用户 |
| FastAPI | 权限校验、意图识别、业务编排、日志 | 不直接开放公网；允许访问指定 LLM API | 非 root 服务用户 |
| 日志/监控 | 保存运行状态和安全事件 | 仅管理员或审计账号可读 | 独立目录和权限 |
| 知识库/数据库 | 保存后续公司资料 | 默认拒绝；按服务账号最小授权 | 独立账号和数据目录 |

Docker 官方文档指出，普通 Docker daemon 通常具有 root 权限；可评估使用 Rootless mode，从而让 daemon 和容器都在非 root 用户命名空间中运行。即使暂不使用 Rootless mode，也不能把普通维护用户随意加入 `docker` 组，因为该组可获得接近 root 的权限。参考：[Docker Rootless mode](https://docs.docker.com/engine/security/rootless/)、[Linux post-installation steps](https://docs.docker.com/engine/install/linux-postinstall/)。

## 4. 用户分区与权限设计

“用户分区”需要同时覆盖服务器账号、应用用户、业务数据和日志权限，不能只创建几个 Linux 文件夹。

### 4.1 服务器操作系统账号

| 账号/角色 | 用途 | 权限要求 |
| --- | --- | --- |
| `sysadmin` | 由公司运维维护系统、网络、补丁和备份 | 可使用受审计的 `sudo`；禁止多人共享账号 |
| `aiagent-deploy` | 发布版本、启停服务、查看必要运行状态 | 仅访问项目部署目录；不默认拥有完整 root 权限 |
| `aiagent-svc` | 实际运行 OpenClaw/FastAPI | 禁止交互登录；无 sudo；只读代码，只写日志和必要数据目录 |
| `aiagent-audit`（可选） | 查看审计日志和运行报表 | 只读日志，无代码修改和服务控制权限 |

建议目录：

```text
/opt/company-ai-agent/       # 部署包和 Compose 配置，deploy 可写，svc 只读
/etc/company-ai-agent/       # 密钥引用和生产配置，仅受权账号可读
/var/lib/company-ai-agent/   # 业务持久化数据，svc 可写
/var/log/company-ai-agent/   # 应用日志，svc 可写，audit 可读
/srv/company-ai-agent-backup/# 加密备份，普通服务账号不可修改
```

### 4.2 钉钉应用用户分区

后端不能只相信消息文本，应使用钉钉返回的稳定用户标识，并在 FastAPI 入口执行“用户 → 角色 → 功能权限”校验。

| 应用角色 | 允许能力 | 禁止能力 |
| --- | --- | --- |
| `employee` | 日报生成、允许范围内的知识问答 | 系统管理、查看他人日志、执行工具写操作 |
| `pilot_user` | 试用新功能并提交反馈 | 修改生产配置 |
| `agent_admin` | 管理白名单、查看脱敏运行状态 | 直接读取 API Key；越权读取业务数据 |
| `blocked`/未知用户 | 只返回无权限提示 | 所有模型、知识库和工具调用 |

生产环境应将 OpenClaw 的 `dmPolicy` 和 `groupPolicy` 从测试期的 `open` 收紧为 allowlist 或公司认可的权限策略，并在 FastAPI 中再次校验，形成两层防护。

### 4.3 数据与会话分区

1. 每次请求保存 `request_id`、钉钉用户 ID、来源、功能名、结果状态和耗时。
2. 会话缓存使用 `tenant/department/user/session` 形式的命名空间，禁止不同用户共用无标识的全局历史。
3. 知识库按部门或资料密级建立集合，检索前先判断用户角色，不在检索后再过滤。
4. 日志默认不记录完整消息、模型密钥、钉钉密钥、认证头和公司敏感文档原文。
5. 如需保留用户输入用于排错，应设置明确保留期限、脱敏规则和访问审计。

## 5. 安全保护方案

### 5.1 主机与 SSH

- 使用公司批准的 Linux LTS 版本，并建立定期安全更新窗口。
- 禁止 root 远程登录；管理员使用个人账号和 SSH Key。
- 关闭不需要的密码登录；重要环境通过公司 VPN、堡垒机或固定管理网段访问。
- 启用主机防火墙，默认拒绝入站，只开放实际需要的端口。
- 保持 AppArmor/SELinux 等强制访问控制机制启用。

Ubuntu 官方建议使用主机防火墙和 OpenSSH 保护网络与远程访问；Ubuntu 默认的主机防火墙管理工具为 UFW。参考：[Ubuntu Security suggestions](https://ubuntu.com/server/docs/explanation/security/security_suggestions/)、[Ubuntu Firewall](https://ubuntu.com/server/docs/how-to/security/firewalls/)、[Ubuntu AppArmor](https://ubuntu.com/server/docs/how-to/security/apparmor/)。

### 5.2 网络

- FastAPI 只监听 `127.0.0.1` 或 Docker 内部网络，不把 `8000` 暴露到公网。
- OpenClaw 只允许访问 FastAPI、钉钉所需域名和必要的 LLM API。
- 后续数据库只接受内部网络连接，并以单独的服务账号认证。
- 如必须提供外部 REST 接口，统一经 Nginx/公司网关使用 HTTPS，并增加认证、限流和请求大小限制。OWASP 建议 REST 服务仅通过 HTTPS 暴露，以保护凭据和数据完整性。参考：[OWASP REST Security](https://cheatsheetseries.owasp.org/cheatsheets/REST_Security_Cheat_Sheet.html)。

### 5.3 密钥和配置

- `DEEPSEEK_API_KEY`、钉钉 `clientSecret` 等不得写入源代码、README、镜像或 Git。
- 测试、预发布和生产环境使用不同密钥，避免共用。
- 生产配置文件设置最小文件权限，并由专用服务身份读取。
- 制定密钥创建、使用、轮换、撤销和泄露应急流程；有公司密钥管理系统时优先接入。
- 日志和报错中主动屏蔽认证头、Token、Cookie、API Key 和 `.env` 内容。

OWASP 建议集中管理、审计和轮换 secrets，并避免把密钥硬编码或提交版本控制。参考：[OWASP Secrets Management](https://cheatsheetseries.owasp.org/cheatsheets/Secrets_Management_Cheat_Sheet.html)、[OWASP Cryptographic Storage](https://cheatsheetseries.owasp.org/cheatsheets/Cryptographic_Storage_Cheat_Sheet.html)。

### 5.4 Agent 与工具调用安全

- Skill 和工具采用白名单；未登记工具不能被模型调用。
- 模型输出不能直接成为 Shell、SQL 或文件路径参数，必须经过结构化校验。
- 写数据库、发通知、修改文件等高风险操作必须二次确认并记录操作者。
- 对单用户、单群和全局设置频率与并发限制，防止 API 费用失控和服务阻塞。
- 对提示词注入、越权索取资料和诱导泄露系统提示的请求返回安全拒绝。
- 模型调用设置超时、最大重试次数、最大输入长度和明确失败提示。

### 5.5 日志、监控与审计

最低记录字段：时间、`request_id`、脱敏用户 ID、入口、功能、权限结果、模型调用结果、耗时、错误类型和版本号。

需要监控：

- `/health` 是否正常；
- 服务/容器是否存活、是否反复重启；
- 错误率、超时率和响应时间；
- DeepSeek API 调用量、Token 或费用异常；
- 未授权访问、连续失败和高风险操作；
- 磁盘与日志目录使用率。

日志需设置轮转和保留期限，安全事件应集中保存且限制访问。参考：[OWASP Logging Cheat Sheet](https://cheatsheetseries.owasp.org/cheatsheets/Logging_Cheat_Sheet.html)。

### 5.6 可用性、备份和回滚

- Compose 为核心服务配置 `healthcheck` 和合适的 `restart` 策略。
- 发布前运行 pytest；发布后依次检查 `/health`、`/version` 和钉钉真实链路。
- 镜像使用明确版本标签，不使用无法追踪的 `latest` 作为唯一发布依据。
- 每次发布保留上一可用版本、配置备份和回滚命令。
- 备份生产配置、权限映射、知识库索引来源和必要业务数据；API Key 应通过密钥系统恢复，而不是散落在普通备份中。
- 定期执行一次恢复演练，确认备份不仅“存在”，而且确实可恢复。

Docker Compose 支持服务健康检查和重启策略，可用于部署后的自动恢复和依赖就绪判断。参考：[Compose services](https://docs.docker.com/reference/compose-file/services/)、[Start containers automatically](https://docs.docker.com/engine/containers/start-containers-automatically/)。

## 7. 当前最优先的下一步

下一步不是立即登录服务器安装软件，而是完成“服务器部署前信息确认”。因为操作系统、网络入口和账号制度会直接决定 Docker、端口、防火墙、用户分区和密钥管理方案。

### 7.1 向导师或公司运维确认

请先收集以下信息：

| 类别 | 需要确认的问题 | 当前值 |
| --- | --- | --- |
| 服务器 | 操作系统及版本是什么？CPU、内存、磁盘多大？ | 待确认 |
| 使用范围 | 是专用服务器、虚拟机，还是与其他业务共用？ | 待确认 |
| 登录 | 通过 SSH、堡垒机还是远程桌面？是否提供个人账号和 sudo？ | 待确认 |
| 网络 | 是否能访问钉钉和 DeepSeek API？是否有代理、出口白名单或固定公网 IP？ | 待确认 |
| 入站 | 是否允许开放端口？钉钉是否确定采用 Stream 模式？ | 待确认 |
| 容器 | 是否已安装 Docker/Compose？公司是否允许使用 Docker？ | 待确认 |
| 域名与证书 | 是否需要域名和 HTTPS 证书？是否已有公司网关或 Nginx？ | 待确认 |
| 用户权限 | 谁负责系统管理、部署、审计和日常使用？是否有统一账号规范？ | 待确认 |
| 数据安全 | 哪些公司资料允许发送给外部 DeepSeek API？哪些数据禁止外发？ | 待确认 |
| 运维要求 | 日志保留多久？如何备份？出现故障通知谁？ | 待确认 |

### 7.2 建议发给导师/运维的说明

> 我已完成 AI Agent 的本地 MVP 和钉钉端到端联调，下一阶段准备部署到公司服务器。为了编写准确的服务器配置与安全方案，需要确认服务器操作系统与资源、登录方式、网络出入站限制、Docker 使用规范、账号权限、日志备份要求，以及哪些业务数据允许调用外部大模型 API。信息确认后，我会先用 mock 模式完成测试部署，再按白名单开放真实 DeepSeek 调用。

### 7.3 信息确认后的第一个开发任务

服务器信息拿到后，立即完成以下四个部署文件，并先在本地或测试机验证：

1. `Dockerfile`：封装 FastAPI 服务，使用非 root 用户运行。
2. `compose.yaml`：编排 FastAPI/OpenClaw、内部网络、健康检查、重启策略和持久化日志。
3. `.env.production.example`：只列变量名和说明，不写真实密钥。
4. `docs/deployment.md`：记录安装、启动、验证、升级、回滚和排错步骤。

## 8. 第一阶段验收标准

部署满足以下条件才算完成：

- [ ] 服务器重启后服务可自动恢复。
- [ ] `GET /health` 和 `GET /version` 返回正常。
- [ ] FastAPI 端口未直接暴露到不可信网络。
- [ ] 服务不以 root 身份运行，部署账号和运行账号分离。
- [ ] 未授权钉钉用户无法触发 DeepSeek、知识库或工具调用。
- [ ] 真实密钥不在 Git、镜像、README 和普通日志中。
- [ ] 一次钉钉日报请求能通过 `request_id` 追踪完整链路。
- [ ] 连续消息、异常输入、模型超时和上游失败有明确处理。
- [ ] pytest、权限测试和端到端测试全部通过。
- [ ] 发布失败时可回滚到上一版本。
- [ ] 已形成部署、运维、安全和故障处理文档。
